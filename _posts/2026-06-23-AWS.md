# AWS Core Services and Architecture Guide

Welcome to the comprehensive guide on core Amazon Web Services (AWS). This reference covers fundamental infrastructure components including networking, compute, storage, databases, and monitoring, supported by detailed architectural diagrams.

---

## 🔐 Security & Identity

### AWS IAM (Identity and Access Management)
AWS IAM enables you to securely manage access to AWS services and resources. It allows you to create and manage AWS users and groups, and use permissions to allow and deny their access to AWS resources.

![AWS IAM Diagram](../assets/images/AWS/IAM.jpg)

* **Users**: Individual people or applications requiring access to AWS.
* **Groups**: Collections of users that share logical permissions.
* **Roles**: Identities with permission policies that can be assumed by trusted entities (e.g., EC2 instances, Lambda functions).
* **Policies**: JSON documents defining permissions (Allow/Deny actions on resources).

## 🌐 Networking & Content Delivery

### AWS VPC (Virtual Private Cloud)
Amazon VPC lets you provision a logically isolated section of the AWS Cloud where you can launch AWS resources in a virtual network that you define.

![AWS VPC Diagram](../assets/images/AWS/VPC.jpg)

* **Subnets**: Ranges of IP addresses in your VPC (Public vs. Private).
* **Internet Gateway (IGW)**: Allows communication between instances in your VPC and the internet.
* **NAT Gateway**: Enables instances in a private subnet to connect to the internet, but prevents the internet from initiating connections with those instances.

### Security Groups
Security Groups act as a virtual firewall for your EC2 instances to control incoming and outgoing traffic.

![AWS Security Groups Diagram](../assets/images/AWS/security_group.jpg)

* **Stateful**: If you send a request from your instance, the response traffic for that request is allowed to flow in regardless of inbound security group rules.
* **Default**: Denies all inbound traffic and allows all outbound traffic.

### Amazon Route 53
Route 53 is a highly available and scalable Domain Name System (DNS) web service.

![Amazon Route 53 Diagram](../assets/images/AWS/route53.jpg)

* **Routing Policies**: Simple, Failover, Geolocation, Geoproximity, Latency, Multi-value answer, and Weighted routing.

### Amazon CloudFront
CloudFront is a fast content delivery network (CDN) service that securely delivers data, videos, applications, and APIs to customers globally with low latency.

![Amazon CloudFront Diagram](../assets/images/AWS/cloudfront.jpg)

* **Edge Locations**: Global network of data centers that cache content close to users.
* **Origin**: The source of truth for your content (e.g., S3 bucket, HTTP server).

## 💻 Compute Services

### Amazon EC2 (Elastic Compute Cloud)
Amazon EC2 provides secure, resizable compute capacity in the cloud. It makes web-scale cloud computing easier for developers.

![Amazon EC2 Diagram](../assets/images/AWS/EC2.jpg)

* **AMIs (Amazon Machine Images)**: Pre-configured templates for instances.
* **Instance Types**: Optimized for Compute, Memory, Storage, or General Purpose.

### Elastic Load Balancing (ELB) & Auto Scaling
ELB automatically distributes incoming application traffic across multiple targets, while Auto Scaling monitors your applications and automatically adjusts capacity.

![Elastic Load Balancing Diagram](../assets/images/AWS/load_balancer.jpg)

![Auto Scaling Diagram](../assets/images/AWS/auto_scaling.jpg)

* **Application Load Balancer (ALB)**: Best suited for HTTP/HTTPS traffic (Layer 7).
* **Network Load Balancer (NLB)**: Best suited for TCP, UDP, and TLS traffic where extreme performance is required (Layer 4).
* **Target Groups**: Logical routing destinations for the Load Balancer.

### AWS Lambda
Lambda lets you run code without provisioning or managing servers. You pay only for the compute time you consume.

![AWS Lambda Diagram](../assets/images/AWS/lambda.jpg)

* **Event-Driven**: Triggered by S3 events, DynamoDB streams, API Gateway, etc.
* **Stateless**: Functions scale rapidly because each execution starts in a fresh environment.

## 📦 Storage & Databases

### Amazon S3 (Simple Storage Service)
Amazon S3 is an object storage service offering industry-leading scalability, data availability, security, and performance.

![Amazon S3 Diagram](../assets/images/AWS/S3.jpg)

* **Buckets**: Logical containers for objects (globally unique namespace).
* **Storage Classes**: Standard, Intelligent-Tiering, Standard-IA, One Zone-IA, Glacier Instant Retrieval, Glacier Flexible Retrieval, Glacier Deep Archive.

### Amazon RDS (Relational Database Service)
Amazon RDS makes it easy to set up, operate, and scale a relational database in the cloud.

![Amazon RDS Diagram](../assets/images/AWS/rds.jpg)

* **Supported Engines**: Amazon Aurora, PostgreSQL, MySQL, MariaDB, Oracle, and SQL Server.
* **Multi-AZ**: Synchronous replication to a standby instance in a different Availability Zone for high availability.
* **Read Replicas**: Asynchronous replication for scaling read performance.

## 📊 Management & Monitoring

### Amazon CloudWatch
CloudWatch monitors your AWS resources and the applications you run on AWS in real time.

![Amazon CloudWatch Diagram](../assets/images/AWS/cloudwatch.jpg)

* **Metrics**: Collects and tracks variables about your resources and applications.
* **Alarms**: Triggers actions based on rules you define.
* **Logs**: Collects, monitors, and analyzes system and application log files.
 
### Amazon SageMaker
Amazon SageMaker is a fully managed service that provides tools to build, train, and deploy machine learning models at scale.

* **Studio**: An integrated development environment for data scientists and ML engineers.
* **Training**: Managed training jobs with built-in algorithms or custom containers.
* **Inference**: Real-time endpoints, multi-model endpoints, and batch transform for model serving.
* **Model Registry**: Track model versions, metadata, and deployment lifecycle.
* **Feature Store**: Centralized repository for ML features used in training and inference.

Example: create a simple SageMaker training job (AWS CLI):

```bash
aws sagemaker create-training-job \
	--training-job-name my-training-job \
	--algorithm-specification TrainingImage=123456789012.dkr.ecr.us-west-2.amazonaws.com/my-algo:latest,TrainingInputMode=File \
	--role-arn arn:aws:iam::123456789012:role/SageMakerRole \
	--input-data-config '[{"ChannelName":"training","DataSource":{"S3DataSource":{"S3DataType":"S3Prefix","S3Uri":"s3://my-bucket/my-training-data/","S3DataDistributionType":"FullyReplicated"}}}]' \
	--resource-config InstanceType=ml.m5.large,InstanceCount=1,VolumeSizeInGB=50
```

SageMaker integrates with S3, IAM, ECR, and CloudWatch and supports distributed training, hyperparameter tuning, and model monitoring for drift and quality.

### Amazon Bedrock — Cheat Sheet for the Exam
Quick, exam-focused notes and a simple architecture diagram to remember core Bedrock concepts.

- **What is Bedrock? (one-liner)**: A managed service that provides access to multiple foundation models (FMs) through a single, consistent API for inference and model customization.

- **When to use Bedrock vs SageMaker**:
	- Use **Bedrock** when you need rapid access to hosted foundation models for text, embeddings, or multimodal inference without managing model infrastructure.
	- Use **SageMaker** when you need full control over model training, distributed tuning, custom containers, or production model lifecycle management.

- **Core exam topics to memorize**:
	- Model invocation flow: Client -> API (Auth/IAM) -> Bedrock model -> Response (optionally stream)
	- Data flow: Input prompts & context often stored in S3; outputs can be logged to CloudWatch or S3 for auditing.
	- Security: IAM policies, KMS encryption (S3/KMS), VPC endpoints (interface endpoints), private network controls.
	- Cost/performance tradeoffs: model selection (small vs large FMs), batching, token window, streaming vs synchronous, regional pricing differences.
	- Customization: lightweight parameter-efficient tuning or prompt engineering + retrieval (RAG) for context augmentation.

- **Operational patterns**:
	- Use caching and short-term embeddings to reduce repeated model calls.
	- Use RAG: index documents (embed -> vector DB), retrieve top-K, inject into prompt window.
	- Add moderation & guardrails: run model outputs through a moderation pipeline before exposing to users.
	- Observability: emit request/response metrics to CloudWatch, sample payloads (PII scrubbed) to S3, track latencies and error rates.

- **Quick CLI / SDK pseudo-example (conceptual)**:
```
# Pseudocode: invoke a Bedrock model (syntax varies by SDK)
client = BedrockClient(auth=iam_role)
response = client.invoke_model(model_id="text-model-1", input_text="Summarize this document: ...", max_tokens=256, temperature=0.2)
print(response.output_text)
```

- **Exam 'cheat' phrases to use in answers**:
	- "Bedrock provides a unified inference API across multiple foundation models."
	- "Use Bedrock for managed FM inference and SageMaker for custom training and MLOps."
	- "Protect data with IAM, VPC endpoints, and KMS; monitor with CloudWatch and SageMaker Model Monitor."

## Practice Questions — Amazon Bedrock (10)
Below are 10 practice questions with concise model answers focused on Bedrock for exam prep.

1. Q: When would you choose Amazon Bedrock over SageMaker for a project?
	- A: Choose Bedrock for managed access to hosted foundation models for inference or embeddings without managing infrastructure; choose SageMaker if you need custom training, distributed tuning, or full MLOps lifecycle control.

2. Q: Name three security controls you would use when integrating Bedrock in a regulated environment.
	- A: Least-privilege IAM roles, KMS encryption for S3 artifacts, and VPC Interface Endpoints to avoid public internet egress.

3. Q: How does Retrieval-Augmented Generation (RAG) reduce hallucinations?
	- A: RAG grounds model outputs with retrieved, high-quality context from a vector DB; better retrieval quality and reranking reduce reliance on model memorized content and lower hallucination risk.

4. Q: What observability metrics are critical for Bedrock-based services?
	- A: Request count, error rate, latency (p50/p95), token usage per request, and downstream vector DB latency; log sampled request/response metadata to S3 for auditing.

5. Q: How can you reduce inference costs when using Bedrock?
	- A: Use smaller models for classification tasks, cache repeated responses, batch requests, and use embeddings + vector DB to reduce token window size.

6. Q: Describe a simple prompt engineering technique to improve model reliability.
	- A: Use system messages and few-shot examples, constrain output formats, and include explicit instructions for fallback behavior when uncertain.

7. Q: What is the tradeoff of streaming responses vs synchronous responses?
	- A: Streaming reduces perceived latency and allows incremental handling, but complicates client-side handling and may not be suitable when the full output must be validated before use.

8. Q: How do you handle PII in prompts sent to Bedrock?
	- A: Scrub or redact PII before sending, store only pseudonymized artifacts, and enforce encryption and access controls on any persisted data.

9. Q: What role does token window size play in Bedrock designs?
	- A: Token window limits how much context you can provide; manage with retrieval (RAG), summarization, or chunking to stay within model limits.

10. Q: Outline a quick validation checklist before deploying a Bedrock-backed endpoint.
	- A: Validate accuracy on a holdout set, run safety/moderation classifiers, measure latency and cost per request, ensure IAM/KMS/VPC controls, and set up monitoring + alerts.

Include these points in architecture explanations and scenario answers — they map directly to common exam prompts.

## Deep Dive: Amazon Bedrock — Exam Subtopics and Cheat Codes
Below are focused subtopics, practical notes, and sample exam questions to help you answer Bedrock-related prompts efficiently.

1) Core Concepts
	- Bedrock is a managed FM inference service that exposes multiple foundation models (owned by AWS partners and AWS) via a single API.
	- Key primitives: model_id, input prompt, model parameters (temperature, top_p, max_tokens), streaming vs sync, embeddings endpoint.
	- Choose model by capability, latency, token window, and cost; always state the tradeoff in answers.

2) Invocation Patterns
	- Synchronous text generation: request->response for small payloads.
	- Streaming responses for long-form generation to reduce perceived latency and begin processing early.
	- Batch inference for large-scale offline generation (e.g., summarize millions of docs) — use async jobs or chunking.
	- Embeddings: call FM embeddings API to get dense vectors for RAG or semantic search.

3) Integration & RAG
	- Typical RAG flow: ingest -> clean -> embed -> store vectors (OpenSearch/Kendra/managed vector DB) -> retrieve top-k -> construct prompt -> call Bedrock.
	- Always include top-k, reranking, and prompt window management in your design to avoid token overflow.

4) Customization & Fine-tuning
	- If exam scenario requires fine-grained control or high-volume production, propose SageMaker for training and containers; otherwise, prefer Bedrock for managed FM inference.
	- Use prompt templates, system messages, and few-shot examples for instruction tuning; consider lightweight adapter techniques when available.

5) Security, Data Protection & Compliance
	- Protect PII: scrub or redact before sending to Bedrock if required by policy; use KMS to encrypt S3 artifacts.
	- Ensure least-privilege IAM roles for any service invoking Bedrock; consider resource-based policies if supported.
	- Use VPC Interface Endpoints and private networking where required by compliance.

6) Observability & Governance
	- Log request/response metadata (not full PII) to S3 and emit metrics to CloudWatch: request_count, error_rate, latency_p50/p95, token_usage.
	- Implement content moderation and rejection thresholds; integrate automated classifiers before exposing outputs.

7) Cost & Performance Tips (cheat codes)
	- Prefer smaller models for classification, embeddings, or short outputs; larger models for complex reasoning.
	- Batch and cache repeated prompts; use embeddings + vector DB to reduce token consumption.
	- Use streaming to improve user-perceived latency while backend finishes generation.

8) Troubleshooting (common exam scenarios)
	- Unexpected high latency: check model choice, region, network (VPC endpoints), cold starts — enable provisioned concurrency if available.
	- High token usage: inspect prompts for verbosity, use truncation/prompt templates, and verify max_tokens parameter.
	- Inconsistent outputs: use deterministic settings (low temperature) for predictable tasks, add examples for few-shot.

9) Sample Exam Questions (short answers)
	- Q: "When would you choose Bedrock over SageMaker?" — A: "Use Bedrock when you need managed, hosted access to foundation models for inference or embeddings without managing model infra; use SageMaker when you need custom training, distributed tuning, or MLOps control."
	- Q: "How to reduce hallucinations in a RAG system using Bedrock?" — A: "Improve retrieval quality (higher-precision vector store, reranking), reduce context noise, use grounding prompts, and post-filter outputs with moderation checks."
	- Q: "List three controls to secure Bedrock integration." — A: "Use least-privilege IAM roles, encrypt data at rest with KMS, and use VPC interface endpoints to avoid public network egress."

10) Quick Study Tasks (hands-on)
	- Build a tiny RAG demo: ingest a few docs to S3, compute embeddings, store in OpenSearch/Kendra, and make Bedrock calls for answer generation.
	- Measure token usage and latency across two models and record tradeoffs in a table for quick reference.

Add these bullets to scenario answers in the exam: always mention tradeoffs, security, monitoring, and how you would validate the model before production.
